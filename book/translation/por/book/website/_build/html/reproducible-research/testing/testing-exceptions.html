Desafios e casos excepcionais em testes

Testando cdigo stochstico

s vezes, o cdigo contm um elemento de aleatoriedade, um exemplo comum de cdigo que faz uso de mtodos Monte Carlo. Testar este tipo de cdigo pode ser muito difcil porque se for executado vrias vezes, gerar diferentes respostas, tudo o que pode estar certo, at mesmo que no contm erros. Existem duas maneiras principais de lidar com o cdigo estocstico de teste:

Usar semente de nmeros aleatrios

Nmero aleatrio de sementes so um pouco difceis de explicar, ento aqui est um exemplo. Aqui est um pouco de script Python que imprime trs nmeros aleatrios.

importar aleatrio

# Imprimir trs nmeros aleatrios
print(random.random())
print(random.random())
print(random.random())

Este script no tem bugs, mas se voc execut-lo repetidamente receber respostas diferentes de cada vez. Agora vamos definir um nmero aleatrio de semente.

importar aleatrio

# Definir uma seed com nmero aleatrio
aleatrio.seed(1)

# Imprimir trs nmeros aleatrios
print(random.random())
print(random.random())
print(random.random()) 
 print(random.random())

Se voc executar este script ele ser executado

0.134364244112
0.847433736937
0.763774618977

e toda vez que voc executar este script voc receber a mesma sada, ele vai imprimir o mesmo trs nmeros aleatrios. Se a semente do nmero aleatrio for alterada, voc receber trs nmeros aleatrios:

0.956034271889
0.947827487059
0.0565513677268

mas de novo voc obter os mesmos nmeros toda vez que o script for executado no futuro.

Sementes numricas aleatrias so uma maneira de tornar as coisas confiavelmente aleatrias. No entanto, um risco com testes que dependem de sementes com nmeros aleatrios que eles podem ser frgeis. Diga que voc tem uma funo estruturada algo como isto:

def minha_function():
  a = calculation_that_uses_two_random_numbers()
  b = calculation_that_uses_cinco_random_numbers()
  c = a + b

Se voc definir a semente de nmero aleatrio voc sempre ter o mesmo valor de c, para que possa ser testada. Mas, diga que o modelo alterado e a funo que calcula um usa um nmero diferente de nmeros aleatrios que ele fez anteriormente. Agora no s um ser diferente, mas b tambm, tambm. porque como mostrado acima os nmeros aleatrios retornados dado um nmero aleatrio de semente est em uma ordem fixa. Como resultado, os nmeros aleatrios produzidos para calcular b tero sido alterados. Isso pode levar a testes falhando quando na verdade no h nenhum erro.

Medir a distribuio dos resultados

Outra maneira de testar o cdigo com uma sada aleatria execut-lo vrias vezes e testar a distribuio dos resultados. Talvez o resultado possa flutuar um pouco, mas espera-se sempre que cerca de 10, dentro de alguma tolerncia. Isso pode ser testado. Quanto mais vezes o cdigo for executado mais confivel a mdia e, portanto, o resultado. No entanto, quanto mais vezes voc executar um trecho de cdigo, mais tempo ele levar seus testes para executar, que pode fazer com que os testes custem tempo proibitivo para poderem conduzir se for possvel obter um resultado fivel. Alm sempre haver um elemento de incerteza e se os nmeros aleatrios carem de uma certa forma, voc pode obter resultado fora da tolerncia esperada, mesmo se o cdigo estiver correto.

Ambas as abordagens para testar o cdigo estocstico podem ainda ser muito teis, mas importante tambm estar ciente das suas potenciais armadilhas.

Testando se os nmeros no-inteiros so iguais

Quando 0.1 + 0.2 no igual a 0.3

H uma complicao com o teste se a resposta um pedao de sadas de cdigo igual resposta esperada quando os nmeros no so inteiros. Vamos ver este exemplo do Python, mas note que este problema no exclusivo do Python.

Se atribuirmos 0.1 a um e 0.2 a b e imprimirmos o seu som, ficaremos 0.3, como esperado.

>>> a = 0.1
>>> b = 0.2
>>> impresso (uma + b)
0.3

Se, no entanto, compararmos o resultado de um mais b com 0.3, ficamos falsos.

>>> print(a + b == 0.3)
Falso

Se ns mostrarmos o valor de um mais b diretamente, podemos ver que existe uma margem de erro sutil.

>>> a + b
0.30000000000000004

Isso porque os nmeros de ponto flutuante so aproximaes de nmeros reais. O resultado dos clculos de pontos flutuantes pode depender do compilador ou interpretador, processador ou arquitetura do sistema e nmero de CPUs ou processos sendo utilizados. Isto pode representar um obstculo importante para a elaborao de testes.

Igualdade em um mundo de ponto flutuante

Quando comparamos nmeros de pontos flutuantes para a igualdade, temos de comparar com uma determinada tolerncia, designada como limiar ou delta. Por exemplo, podemos considerar os valores calculados e esperados de algum nmero como iguais se o valor absoluto de sua diferena estiver dentro do valor absoluto de nossa tolerncia.

Muitos quadros de ensaio proporcionam funes para comparar a igualdade dos nmeros de pontos flutuantes com uma determinada tolerncia. Por exemplo, para o framework pytest:

import pytest

a = 0,1
b = 0{,}2
c = a + b
assert c == pytest.approx(0{,}3)

isso passa, mas se os 0,3 fossem alterados para 0,4 falharia.

Quadros de teste unitrios para outras lnguas tambm frequentemente fornecem funes semelhantes:

  • Cunit for C: CU_ASSERT_DOUBLE_EQUAL(real, esperado, granularidade)

  • CPPUnit for C++: CPPUNIT_ASSERT_DOUBLES_EQUAL(esperado, atual, delta)

  • googletest para C++: ASSERT_NEAR(val1, val2, abs_error)

  • FRUIT para Fortran: subrotinine assert_eq_double_in_range_(var1, var2, delta, mensagem)

  • JUnit for Java: org.junit.Assert.assertEquals(double expected, double real delta)

  • teste para R:

    • expect_equal(atual, esperado, tolerance=DELTA) - erro absoluto dentro do DELTA

    • expect_equal(real, esperada, scale=expected, tolerance=DELTA) - erro relativo dentro do DE T A
