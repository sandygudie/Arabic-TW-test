Por que contineres?

Mesmo para projetos moderadamente complexos, o tamanho da pilha de dependncias de software pode ser enorme. Faa um pipeline simples para criar um relatrio pdf para uma anlise escrita em R usando Rmarkdown, por exemplo. Fazer isto reprodutvel, no apenas (i) os respectivos pacotes R precisam ser instalados e (ii) a verso R precisa ser a mesma, mas tambm (iii) as verses do pandoc e LaTeX precisam ser as mesmas que durante o tempo de execuo.

Em vez de tentar resolver essas dependncias por meio de um gerenciador de pacotes (como conda) - que tambm depende de todos os softwares necessrios estarem disponveis em um nico gerenciador de pacotes - pode ser mais fcil criar um snapshot de todo o ambiente de computao, incluindo todas as dependncias. Estes ambientes computacionais so ento auto-contidos, da a designao de contineres.

O que so contineres?

Os contineres permitem que um pesquisador empacote um projeto com todas as partes necessrias - como bibliotecas, dependncias e configuraes do sistema - e enviar tudo como um pacote. Qualquer um pode ento abrir um continer e trabalhar dentro dele, ver e interagir com o projeto como se a mquina a que eles esto acessando fosse idntica mquina especificada no continer - independentemente de qual seu ambiente computacional realmente. Eles so projetados para facilitar a transferncia de projetos entre ambientes muito diferentes.

De certa forma, os contentores se comportam como uma mquina virtual. Para o mundo exterior, parecem ter o seu prprio sistema completo. No entanto, ao contrrio de uma mquina virtual, ao invs de criar todo um sistema operacional virtual, alm de todas as ferramentas e software tipicamente empacotados com um, Os recipientes s contm os componentes individuais de que eles precisam para operar o projeto que eles contm. Isso d um aumento significativo de desempenho e reduz o tamanho do aplicativo.

Os contineres so uma maneira particularmente til de reproduzir a pesquisa que depende de software para ser configurado de certa maneira. ou que faz uso de bibliotecas que variam entre (ou no existem) diferentes sistemas. Em resumo, contineres so uma forma mais robusta de compartilhar pesquisas reprodutveis do que sistemas de gerenciamento de pacotes ou Binder porque eles reproduzem todo o sistema usado para a pesquisa No apenas os pacotes usados explicitamente por ele. O seu principal reverso o que se deve sua maior profundidade, so conceitualmente mais difceis de entender e produzir do que muitos outros mtodos de replicao de ambientes computacionais.

Ben Corrie d uma viso geral razoavelmente acessvel dos conceitos principais em O que um continer?.

O que so Imagens?

As imagens so os arquivos usados para gerar contineres. Os seres humanos no fazem imagens; escrevem receitas para gerar imagens. Os contineres so, em seguida, cpias idnticas instanciadas a partir de imagens.

Pense nisso assim:

  • Um arquivo de receita que uma escrita humana contm todas as etapas para gerar uma verso funcional do projeto e seu ambiente computacional. mas no h materiais reais. Pense nisso como um planeta.

  • Construir uma imagem leva essa receita e usa-a, monta todos os pacotes, bibliotecas de software e configuraes necessrias para fazer o projeto e o ambiente de pleno direito e empacot-los de forma condensada. Pense em imagens como um pedao de moblia de pacote fixo feita usando o diagrama.

  • Os contineres levam essa imagem e montam uma verso totalmente funcional do projeto e do ambiente necessrio para execut-la. Pense nisto como a montagem dos mveis de pacote plano.

Portanto, se um pesquisador quiser permitir que outros reproduzam seus trabalhos, eles precisariam escrever um arquivo de receita e us-lo para construir uma imagem do projeto deles. Em seguida, eles podem compartilhar este arquivo de imagem com qualquer pessoa que queira replicar seu trabalho. Essa pessoa pode ento usar a imagem para gerar um continer contendo uma verso de funcionamento do projeto.

No Docker, os arquivos de receita usados para gerar imagens so conhecidos como arquivos Dockerfiles e devem ser nomeados como arquivo Dockerfile.

Comandos da tecla

Aqui esto alguns comandos chave para criar e trabalhar com contineres:

  • Para construir uma imagem de um arquivo Dockerfile, v para o diretrio onde o arquivo Dockerfile e executado:

    sudo docker build --tag image_name .

  • Para listar as imagens em seu sistema, use:

    imagens docker sudo

  • Para remover uma imagem, execute:

    sudo docker rmi image_name

  • Para abrir um continer de uma imagem, execute:

    sudo docker executa -i -t image_name

    O -i -t flags abrem automaticamente um terminal interativo dentro do continer para que voc possa ver e interagir com os arquivos do projeto.

  • Para sair de um terminal interativo, use:

    saindo

  • Para obter uma lista de contineres ativos com IDs, execute:

    recipientes docker sudo

  • H tambm trs comandos principais usados para alterar o status dos contineres:

    • Pausing suspende o processo rodando o continer.

      sudo docker pause container_ID

      Os contineres podem ser despausados substituindo pause por cancelar.

    • Parar um continer termina o processo que o executa. Um recipiente deve ser interrompido antes que possa ser excludo.

      sudo docker stop container_ID

      Um continer parado pode ser reiniciado substituindo stop com restart.

    • Se parar no funcionar com contineres pode ser morto usando

      sudo docker kill container_ID

  • Para remover um continer, execute:

    sudo docker rm container_ID

MUNDO

Esse comando pode ser usado no Dockerfiles para alterar o diretrio de trabalho atual. Os comandos que seguem isso no arquivo Dockerfile sero aplicados dentro do novo diretrio de trabalho a no ser / at que outro WORKDIR mude o diretrio de trabalho. Quando um recipiente for aberto com um terminal interativo, o terminal ser aberto no diretrio de trabalho final. Aqui est um exemplo simples de um arquivo Dockerfile que usa WORKDIRe o continer que ele gera.

# Configurao bsica
FROM ubuntu
Atualizao do apt-get RUN

# Faa um diretrio chamado
RUN mkdir A

# Faa o diretrio de trabalho A
WORKDIR A

# Faa dois diretrios, um chamado B_1 e um chamado B_2
RUN mkdir B_1
RUN mkdir B_2

Diretrios B_1 e B_2 foram criados dentro do diretrio A.

O WORKDIR deve ser usado quando alterar diretrios necessrio durante a construo de uma imagem. Pode ser tentador usar o directory_name da RUN cd em vez disso. j que esta sintaxe ser mais familiar para aqueles que normalmente funcionam atravs da linha de comando, mas isso pode levar a erros. Aps cada instruo RUN em um arquivo Dockerfile, a imagem salva e todos os seguintes comandos so aplicados novamente imagem. As an example, here is what happens in the above example if the WORKDIR A line is swapped for RUN cd A.

Todos os diretrios esto no nvel superior neste caso, em vez de B_1 e B_2 estando dentro de A. Isso porque a imagem foi reiniciada aps o comando RUN cd e aberta no nvel superior (root) por padro, portanto, foi a que os comandos mkdir B_1 e mkdir B_2 entraram em vigor.

Construindo imagens e .dockerignore Arquivos

Como mencionado na seo de comandos chave de , Para construir uma imagem, abra um terminal no mesmo diretrio que o arquivo Dockerfile a ser usado e executado:

sudo docker build --tag name_to_give_image .

Quando uma imagem construda tudo no diretrio do Dockerfile e abaixo (isso chamado de contexto) enviado para o Docker daemon para construir a imagem. O daemon usa o Dockerfile e seu contexto para construir a imagem. Se o contexto contm muitos arquivos grandes, que no so necessrios para a construo da imagem, (dados antigos, por exemplo), ento uma perda de tempo envi-los para o daemon. Fazer isso pode deixar o processo de construo de uma imagem mais lento. Os arquivos podem ser excludos do contexto listando-os em um arquivo de texto chamado .dockerignore. uma boa prtica faz-lo.

Os arquivos no precisam ser listados individualmente no arquivo .dockerignore. Aqui est um exemplo do contedo de um arquivo .dockerignore:

*.jpg
**/*.png
data_files/*
file_to_exclude.txt

Isso exclui do contexto:

  • Todos os arquivos .jpg no mesmo diretrio que o arquivo Dockerfile

  • Todos os arquivos .png no mesmo diretrio do arquivo Dockerfile ou quaisquer subdiretrios dentro dele

  • Todos os arquivos dentro do diretrio data_files

  • O arquivo chamado file_to_exclude.txt

Compartilhando imagens

Imagens Docker podem ser compartilhadas mais facilmente via Docker Hub, o que requer uma conta. Diga a dois investigadores, Alice e Bob, que esto colaborando num projeto e Alice deseja compartilhar uma imagem de algum do seu trabalho com Bob.

Para fazer isso, Alice deve:

  • Escreva um arquivo Dockerfile para produzir uma imagem do seu trabalho.

  • Construa a imagem. Ela (sendo inventiva) a chama de image_name

  • V para DockerHub e cadastre-se para uma conta. Diga Alice (novamente, estando inventivo) escolhe o nome de usurio username_Alice

  • Entrar no DockerHub atravs do terminal em sua mquina usando:

    login do sudo docker

  • Marque a imagem de seu projeto em sua mquina atravs da linha de comando fornecendo o nome da imagem e usando o padro username/image_name:version. Ento o Alice executa o comando:

    sudo docker tag image_name username_Alice/image_name:version_1

  • Faa push da imagem para sua conta do Docker Hub usando:

    sudo docker tag push username_Alice/image_name:version_1

  • A imagem do Alice est online e pode ser baixada. Acima para o Bob

Bob (supondo que ele j tenha instalado o Docker) pode abrir um continer da imagem do Alice simplesmente executando

docker sudo run -i -t username_Alice/image_name:version_1

Inicialmente, o Docker ir procurar essa imagem na mquina do Boi. Quando ele no encontr-lo, automaticamente buscar o DockerHub, Baixe a imagem do Alice e abra o continer com o trabalho e o ambiente do Alice na mquina do Box.

Copiando arquivos para e de contineres

Os contineres agem muito como mquinas virtuais; como resultado, copiar arquivos para e fora deles no to trivial como copiar arquivos para diferentes locais dentro do mesmo computador.

Um arquivo pode ser copiado da mquina executando um continer no continer usando:

sudo docker cp file_name container_ID:path_to_where_para_o_arquivo_do_arquivo/arquivo

Lembre que os IDs de continer podem ser obtidos usando sudo docker contineres.

Um arquivo pode ser copiado de dentro de um continer para a mquina executando o seguinte comando na mquina que executa o continer:

sudo docker cp container_ID:path_to_o_caminho_de_arquivo/arquivo _para_onde_do_arquivo_para_o_arquivo_nome_de_arquivo

Se a segunda parte (o path_to_where_to_put_file/file_name) substitudo por um ., em seguida, o arquivo ser copiado para qualquer diretrio que o terminal esteja executando o comando.

Volume

Toda vez que um recipiente aberto a partir de uma imagem, esse recipiente completamente novo. Diga que um recipiente est aberto e que feito trabalho dentro dele. Se esse continer fechado, e a imagem de que veio usada novamente para iniciar outro continer, nenhum desse trabalho ser no novo. Ter simplesmente o estado de partida descrito na imagem.

Isso pode ser um problema se um pesquisador quiser trabalhar em um recipiente ao longo do tempo. Felizmente, h uma maneira de contornar esta situao utilizando volumes. Os volumes armazenam o trabalho feito em um recipiente mesmo depois de fechado, e podem ser usados para carregar que funcionam em futuros contineres.

Para criar/usar um volume, execute:

sudo docker run -i -t --mount source=volume_name,target=/target_directory image_name

Voc deve dar ao seu volume um nome mais descritivo que o volume_name. Um diretrio de destino necessrio; s funciona dentro deste diretrio ser salvo no volume. Assim que o pesquisador estiver pronto, ele poder fechar o recipiente normalmente. Quando eles voltam para o projeto e querem continuar o seu trabalho, eles s precisam usar o mesmo comando que acima, e ir carregar o trabalho contido no volume_name no novo continer. Tambm a se salvar qualquer novo trabalho.

Abaixo est uma lista de comandos relacionados ao volume:

  • Para listar volumes: sudo docker volume ls

  • Para excluir um volume: sudo docker volume rm volume_name

  • Para excluir todos os volumes no anexados: sudo docker volume prune

Se ao excluir um continer, um -v includo aps rm no sudo docker rm container_ID, qualquer volume associado ao continer tambm ser excludo.

Singularidade

Pr-requisitos: No momento, Singularity s executado em sistemas Linux (por exemplo, Ubuntu). Se voc usar o macOS, Singularity Desktop para macOS est no estgio de lanamento Beta.

Uma desvantagem significativa do uso do Docker para pesquisa reprodutvel que no se pretende como uma aplicao do user-space mas como uma ferramenta para administradores de servidores. Como tal, requer acesso root para operar. No entanto, no h qualquer razo para que a execuo de uma anlise exija o acesso de raiz do utilizador. Isto especialmente importante quando os clculos so conduzidos em um recurso compartilhado, como sistemas de HPC onde os usurios nunca tero acesso root.

O software de continer de singularidade foi introduzido para resolver este problema. Singularidade foi criada com sistemas HPC e pesquisa reprodutvel em mente (consulte [this](https://www.youtube.com/watch?v=DA87Ba2dpNM). Isso no requer acesso root para executar (apenas para construir o continer imagens! , e assim permite que usurios de HPC criem imagens de recipientes localmente antes de executar anlises em um grupo de alto desempenho, por exemplo. Como benefcio adicional, Isso torna possvel o uso de quase qualquer software em um sistema HPC sem ter que incomodar a equipe de administrao com a instalao.

Furthermore, since Docker is the most well-known containerization approach, singularity aims at maintaining compatibility with docker containers. Isto significa que a singularidade pode ser usada para executar contineres docker normais (sem exigir acesso root!).

Singularidade pode ser usada para executar imagens Docker ou estend-las construindo novas imagens com base em docker containers como camada base. Por exemplo, poderamos usar a singularidade para criar um recipiente de ubuntu vanilla com um casco usando a imagem docker de ubuntu:

shell de singularidade docker://ubuntu

(digite exit para deixar o shell interativo novamente).

Assim como as imagens docker so construdas usando o arquivo Dockerfile arquivos, contineres singularidade so construdos a partir de arquivos de definio de singularidade. O processo e a sintaxe so semelhantes aos arquivos docker, mas existem diferenas subtis. Como um exemplo mnimo de funcionamento, podemos construir um continer lolcow com base na imagem oficial do docker do ubuntu. Coloque o seguinte em um arquivo lolcow.def (com base na Documentao do Singularity):

Bootstrap: docker
From: ubuntu

%pperde
    apt-get -y update
    apt-get -y install fortune cowsay lolcat

%environment
    export LC_ALL=C
    export PATH=/usr/games:$PATH

%runscript
    fortune lolcat

Esta receita usa uma imagem docker como base (ubuntu), instala alguns pacotes apt e modifica algumas variveis de ambiente, e especifica o runscript (que executado usando o comando singularidade run). Detalhes sobre o formato do arquivo de definio de singularidade podem ser encontrados na documentao oficial.

Uma imagem de continer pode ser construda (exigindo root!) via:

sudo singularidade construir lolcow.simg lolcow.def

Isto ir puxar a imagem ubuntu do DockerHub, executar as etapas da receita no arquivo de definio e produzir um nico arquivo de imagem de sada (lolvaca. img). Finalmente, o script `` executado como

Armazenamento de longo prazo de imagens de continer

importante notar que um simples arquivo de receitas de continer no reproduzvel em si, uma vez que o processo de construo depende de vrias fontes (on-line). Assim, o mesmo arquivo de receita pode levar a imagens diferentes se as fontes subjacentes forem atualizadas.

Para alcanar a reprodutibilidade verdadeira, importante armazenar o continer imagens. Para imagens singulares, isso particularmente fcil, j que uma imagem simplesmente um arquivo grande. Estas podem variar em tamanho, de algumas dezenas de megabytes (microcontainers) a vrios gigabytes, e, portanto, no so adequados para ser armazenado em um repositrio git em si Um repositrio gratuito, soluo elegvel e de longo prazo para armazenar imagens contines zenodo. rg que permite at 50 Gb por repositrio. Desde que zenodo mints DOIs para todo o contedo enviado, as imagens so imediatamente referenciveis. Em contraste com o Docker Hub (que tambm aceita apenas imagens docker), zenodo tambm est claramente orientado para o armazenamento a longo prazo e para a descoberta atravs de um sistema de metadados sofisticado. Assim, idealmente adequado para armazenar contentores cientficos associados a anlises especficas, uma vez que estas tendem a no mudar com o tempo.

Palavras de Aviso

Embora a singularidade e o docker possam parecer semelhantes, so conceitualmente muito diferentes. Alm do fato bvio que a singularidade no requer acesso root para contineres executados, lida tambm com a distino entre o sistema de arquivos de host e o de continer de forma diferente. Por exemplo, por padro, a singularidade inclui alguns pontos de bind no container, nomeadamente:

  • $HOME

  • /sys:/sys

  • /proc:/processo

  • /tmp:/tmp

  • /var/tmp:/var/tmp

  • /etc/resolv.conf:/etc/resolv.conf

  • /etc/passwd:/etc/passwd

  • $PWD

Nota, $PWD til, uma vez que implica que todos os arquivos no diretrio de trabalho so visveis dentro do container. Vinculando $HOME por padro, no entanto, tambm implica que o software que usa arquivos de configurao de $HOME pode se comportar inesperadamente, uma vez que os arquivos de configurao especficos da imagem so substitudos pelas configuraes de usurios atuais em $HOME. Embora este comportamento seja til em cenrios HPC, potencialmente perigoso para a investigao reprodutvel. Para evitar problemas em potencial, qualquer software instalado em um continer de singularidade deve ser apontado para um arquivo de configurao global, independente do usurio.
